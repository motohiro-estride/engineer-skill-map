# v1 データモデル仕様

engineer-skill-map v1 の論理データモデル。元データ (`private/` 配下の Excel 2ファイル) を統合し、公開Web / Excel出力 / ローカル(自分用) の3コンテキストで参照される。

## 設計の前提

- **公開Web**: GitHub Pages で `/` に統合版を1サイト公開
- **Excel出力**: 同一データから生成、営業/面談者へファイル添付で渡す
- **ローカル(private)ビルド**: 自分用の閲覧、面談時の手元参照、メモ機能を含む
- v1 では Profile（ターゲット別シート分割）機能は持たない。**v2 で追加予定** （詳細は末尾「将来拡張枠」）

---

## 表示レベル (3段階)

各フィールドは下記いずれかのレベルに属する:

| レベル | 公開Web | Excel | ローカル | 例 |
|---|---|---|---|---|
| **全公開** | ○ | ○ | ○ | 自己PR、技術スタック、プロジェクト概要 |
| **Excel-only** | ✕ | ○ | ○ | 氏名、最寄駅、勤務開始可能日、生年月日 |
| **自分のみ** | ✕ | ✕ | ○ | private_memo (Project / TechTag) |

実装方針: `private/` 配下のデータを公開Webビルドでは読み込まないことで、情報漏洩リスクをゼロにする。

### visibility ステータス (3値)

| visibility | Web | Excel | 集計に含む | 詳細ページ | ローカル |
|---|---|---|---|---|---|
| **default** (デフォルト) | ○ | ○ | ✓ | ○ | ○ |
| **stats_only** | ✕ | ✕ | ✓ | ✕ (404) | ○ |
| **archived** | ✕ | ✕ | ✕ | ✕ (404) | ○ |

- **default**: 通常案件
- **stats_only**: 表に出したくないが経験としてカウントしたい案件 (例: NDA 厳しめだが技術的経験は計上したい)
- **archived**: 完全に外したい案件 (例: 新人OJT、技術的価値の薄いもの)

ローカル(private)ビルドでは、status に関わらず全件表示。自分の棚卸し用途のため。

## TechTag

| フィールド | 型 | 表示レベル | メモ |
|---|---|---|---|
| 名前 | string | 全公開 | 表示用。バージョンも別タグとして登録 (例: `React`, `React 18`, `React 17`) |
| baseName | string | 全公開 | フィルタ集約用 (例: `React 18` の baseName は `React`) |
| カテゴリ | enum | 全公開 | Language / Framework / OS / Middleware / Database / Tool / Cloud |
| 評価レベル | enum {A, B, C, D, E} | 全公開 | MM の保有技術一覧由来 |
| 累積経験年数 | float (derived) | 全公開 | 関連 Project の期間合計。`archived` 除外、`stats_only` 含む |
| 関連プロジェクト件数 | int (derived) | 全公開 | 同集計ルール |
| private_memo | text | 自分のみ | 「Reactと聞かれたら〜」のような切り口別カンペ |

評価レベルの定義 (MM 由来):
- **A**: 業務の独力遂行。業務課題発見・解決。後進教育
- **B**: 業務の独力遂行
- **C**: 業務を上位者指導のもと遂行
- **D**: 実務を通じた学習経験あり
- **E**: 学習経験あり

| 派生フィールド | 算出元 | ルール |

## 将来拡張枠 (v2 以降の検討材料)

v1 のスキーマを非破壊で拡張可能な形で残しておくべきもの:

### 1. Output Profile (シートのバリエーション)

ターゲット (案件種別/業界/読者) に応じた **複数バリエーションのスキルシート** を出し分ける機能。

- 新エンティティ `OutputProfile` を導入 (id, name, description)
- `Project` に `profiles: OutputProfile[]` フィールドを追加 (デフォルト空配列 = 全 Profile に表示、非破壊変更)
- 公開Web は Profile ごとに別 URL を生成、Excel はビルド時に Profile 指定
- v1 ではこの機能を持たないが、データモデル設計はこの拡張を想定して妥協していない

#### v2 で要再議論となる派生事項

- **集計ポリシー**: Profile 単位の集計か全件横断か (v1 の方針は「全件横断」= 案 a)
- **配布運用**: GitHub Pages は認証なしのため、URL 体系をどう設計するか
  - A. 運用で割り切る / B. 推測困難な suffix / C. JSパスワードゲート / D. Cloudflare Pages + Access に乗り換え / E. Web は default のみ、特化版は Excel/PDF のみで配布
  - v1 では default 1本のみのため問題は発生しない
- **Profile の管理粒度**: v2 で実装するときに「最初に何個 Profile を作るか」も決める

### 2. 案件管理機能

営業から紹介された案件、面談スケジュール、結果トラッキングを扱う「自分専用」機能。スキルシートとは別レイヤだが、データソース (技術タグ・業界タグ) は共通利用したい。

- v1 は GitHub Pages 静的サイト前提で対応不可
- v2 で改めて、A. localStorage / B. GitHub をDB / C. Pages を捨ててバックエンド導入 / D. 外部SaaS 連携 のいずれかを選択
